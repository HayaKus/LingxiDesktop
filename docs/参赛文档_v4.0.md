# 一、背景

2025年，AI Agent百花齐放：Claude Code写代码、GitHub Copilot补全代码、ChatGPT对话、豆包视频通话。

但我发现一个问题：**虽然专业，但不够"快"。**

**什么叫"不够快"？**

钉钉上有人问："上周需求进度怎么样？"
你的大脑产生疑问："需求现在什么状态？"
但你还需要：截图→打开AI→上传→输入→等待→手动查Aone→整理→回复

**从看到问题到AI给答案，中间隔了7步操作。**

**我想要的是**：当我看到问题、产生想法时，AI就已经开始工作，而不是让我先"转述"一遍。

就像豆包的视频通话，AI直接"看到"你周围环境，你问"这是哪里"就得到答案，**桌面端也应该如此。**

## 4.2 核心能力实现

### 4.2.1 桌面视觉感知

#### 屏幕截图能力

**技术实现**：
- 使用 Electron desktopCapturer API 获取屏幕源
- 支持窗口和屏幕两种截图模式
- 默认分辨率 1920x1080

**智能窗口识别**：
- 自动排除灵析自身窗口
- 优先选择最前端活跃窗口
- 支持多显示器场景

#### 剪贴板图片监控

**技术实现**：
- 实时监听系统剪贴板消息
- 自动检测图片内容
- 支持多种图片格式

**应用场景**：
- 用户截图后自动识别
- 支持从其他应用复制图片
- 无需手动上传
- 自动压缩优化

#### 图片优化策略

**压缩算法**：
- 格式转换：PNG → JPEG（质量80%）
- 尺寸缩放：4K → 1080p
- 使用sharp库进行高效处理

**优化效果**：
- 原始大小：6MB (4K PNG)
- 优化后：300KB (1080p JPEG)
- 压缩率：95%
- 视觉质量：几乎无损

### 4.2.2 MCP集成

Model Context Protocol（模型上下文协议），是2024年11月由Anthropic发布的AI调用外部系统的标准化协议，被誉为"AI的USB接口"

#### 技术实现

**SDK集成**：
- 使用官方 @modelcontextprotocol/sdk（github链接：https://github.com/modelcontextprotocol#getting-started）
- 通过HTTP请求MCP容易踩坑，建议用MCP SDK进行请求。

**OAuth 2.1 PKCE认证流程**：

Aone开放平台上的MCP遵循是2025-06-18版本MCP协议。

协议链接：https://modelcontextprotocol.io/specification/2025-06-18/basic/authorization

2. AI决策调用哪个工具

### 4.2.4 上下文管理

上下文管理是灵析的核心技术之一，通过智能的Token预计算、滑动窗口机制和差异化保留策略，在千问VL的131K超大上下文窗口下实现了用户对上下文容量无感的显著效果。

#### Token预计算机制

**精确计算**：
- **文本Token计算**：使用tiktoken库（OpenAI官方分词器）进行精确计算，支持中英文混合文本
- **图片Token计算**：基于图片分辨率动态计算，公式为 `(width × height) / (512 × 512) × 170`
- **实时监控**：每次对话前预计算总Token使用量，确保不超过模型限制

**预计算流程**：
```
发送请求前
   ↓
1. 计算系统提示词Token（固定）
   ↓
2. 计算MCP工具描述Token（固定）
   ↓
3. 遍历历史消息，累加Token
   - 文本消息：tiktoken计算
   - 图片消息：分辨率公式计算
   ↓
4. 计算当前用户输入Token
   ↓
5. 预留AI响应空间（~64K tokens）
   ↓
6. 判断是否超限
   - 未超限：直接发送
   - 超限：触发智能裁剪
```

**Token预算分配**：

| 组成部分 | Token分配 | 占比 | 说明 |
|---------|----------|------|------|
| 系统提示词 | ~2,000 | 1.5% | 固定，定义AI角色和能力 |
| MCP工具描述 | ~5,000 | 3.8% | 固定，48+工具的完整描述 |
| 历史对话 | ~50,000 | 38.1% | 动态，根据实际对话调整 |
| 当前输入 | ~10,000 | 7.6% | 动态，用户当前问题+图片 |
| 图片内容 | ~10,000 | 7.6% | 动态，当前截图或剪贴板图片 |
| AI响应预留 | ~54,072 | 41.2% | 预留，确保AI有足够空间回答 |
| **总计** | **131,072** | **100%** | 千问VL Max上下文窗口 |

#### 滑动窗口机制

**核心思想**：
- 采用滑动窗口（Sliding Window）策略管理历史对话
- 始终保留最新的N轮对话，移除最早的对话
- 窗口大小动态调整，基于Token使用量而非固定轮数

**滑动窗口工作原理**：
```
初始状态（3轮对话）：
[消息1] [消息2] [消息3] [消息4] [消息5] [消息6]
 ↑────────────────窗口────────────────↑

新消息到来：
[消息1] [消息2] [消息3] [消息4] [消息5] [消息6] [消息7] [消息8]
                 ↑────────────────窗口────────────────↑
                 
窗口滑动（移除最早消息）：
        [消息3] [消息4] [消息5] [消息6] [消息7] [消息8]
         ↑────────────────窗口────────────────↑
```

**动态窗口大小**：
- 不固定保留轮数，而是基于Token预算
- 文字对话多时：可保留更多轮（5-8轮）
- 图片对话多时：保留较少轮（2-4轮）
- 自适应调整，最大化利用上下文空间

**窗口边界保护**：
- **最新消息保护**：最近1轮对话永不移除
- **系统消息保护**：系统提示词永不移除
- **工具结果保护**：最近的工具调用结果优先保留
- **关键信息保护**：用户明确引用的历史信息标记为重要

#### 图片与文字差异化保留策略

这是灵析的核心创新，通过差异化对待图片和文字，实现了对话历史提升67%的突破。

**策略原理**：
- **图片特点**：Token消耗大（单张~2000-5000 tokens），但信息密度相对较低
- **文字特点**：Token消耗小（单条~50-200 tokens），但承载关键上下文
- **核心策略**：优先移除历史图片，保留文字对话

**三级保留优先级**：

**第一优先级（永不移除）**：
- 系统提示词
- MCP工具描述
- 最新一轮对话（包括图片）
- 当前用户输入

**第二优先级（重要保留）**：
- 最近3轮的文字对话
- 最近的工具调用结果
- 用户明确引用的历史信息

**第三优先级（可移除）**：
- 历史图片（优先移除）
- 早期文字对话（按需移除）

**智能裁剪算法**：
```
裁剪流程：
1. 计算当前总Token使用量
   ↓
2. 判断是否超过限制（131K - 64K预留 = 67K）
   ↓
3. 如果超限，执行裁剪：
   
   步骤1：移除最早的历史图片
   - 从最早的对话开始遍历
   - 找到包含图片的消息
   - 仅移除图片，保留该消息的文字部分
   - 重新计算Token
   - 如果仍超限，继续移除下一张图片
   
   步骤2：如果所有历史图片都已移除仍超限
   - 移除最早的完整对话轮次
   - 保留最近3轮对话不动
   - 重新计算Token
   - 如果仍超限，继续移除
   
   步骤3：极端情况处理
   - 如果移除到只剩3轮对话仍超限
   - 压缩当前输入图片（降低分辨率）
   - 或提示用户简化问题
```

### 4.2.5 模型降级策略

#### 多模型支持

灵析支持多个千问VL系列模型，通过智能降级策略确保服务的高可用性。

**模型配置**：

| 模型名称 | 类型 | 上下文 | 优先级 | 说明 |
|---------|------|--------|--------|------|
| qwen-vl-max-latest | 多模态 | 131K | 1 | 最新版本，性能最优 |
| qwen-vl-max | 多模态 | 131K | 2 | 稳定版本 |
| Qwen-VL | 多模态 | 131K | 3 | 基础版本 |
| qwen3-vl-plus | 多模态 | 131K | 4 | Plus版本 |
| qwen-vl-max-inc | 多模态 | 131K | 5 | 内部版本 |
| qwen-vl-plus-inc | 多模态 | 131K | 6 | Plus内部版本 |

**模型选择策略**：
- 默认使用优先级最高的可用模型
- 所有模型均支持多模态（图片+文字）
- 所有模型均支持131K超大上下文
- 根据可用性和性能动态选择

#### 降级策略

**触发条件**：
- API调用失败（网络错误、服务不可用）
- 限流错误（429 Too Many Requests）
- 模型暂时不可用（503 Service Unavailable）
- 上下文超限（虽然都是131K，但实际可能有差异）

**降级流程**：

依次使用上述6个千问VL模型，直至请求成功或无可用模型。

**智能降级特性**：
- **自动切换**：检测到失败后自动尝试下一个模型
- **快速失败**：单个模型失败后立即切换，不等待超时
- **状态记录**：记录每个模型的可用状态，优先选择可用模型
- **用户透明**：降级过程对用户透明，无需手动干预

### 4.2.6 鉴权管理

#### BUC登录

**技术实现**：
- 使用Electron BrowserWindow打开登录页面
- 监听登录成功回调
- 提取并保存Token
- 自动关闭登录窗口

**Token管理**：
- 本地加密存储
- 自动刷新机制
- 过期自动重新登录

#### MCP OAuth鉴权

**OAuth 2.1 PKCE流程**：
1. 生成PKCE参数（Code Verifier和Challenge）
2. 构建授权URL
3. 打开授权页面
4. 监听回调获取Authorization Code
5. 用Code + Verifier换取Token
6. 保存Access Token和Refresh Token

**Token自动刷新**：
- 检测Token是否即将过期
- 使用refresh_token获取新的access_token
- 自动更新本地存储

**安全特性**：
- PKCE防止授权码拦截攻击
- State参数防止CSRF攻击
- Token本地加密存储
- 自动刷新保证持续可用
