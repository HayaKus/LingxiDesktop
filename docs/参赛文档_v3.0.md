# 一、背景

2025年真的是AI Agent百花齐放的一年，各类AI工具层出不穷，功能都非常专业、强大：
- **Claude Code**：能帮你写代码、改bug、查文档
- **GitHub Copilot**：代码自动补全，智能建议
- **ChatGPT/Claude**：强大的对话能力，无所不能
- **豆包/通义千问**：移动端支持视频通话，实时看到你周围的环境

但在实际使用中，我发现了一个核心问题：**这些AI Agent虽然专业，但不够"快"。**

## 什么叫"不够快"？

不是响应速度慢，而是**从看到问题到AI开始工作的链路太长**。

举个例子：
- 钉钉上有人问你："上周的需求进度怎么样？"
- 你的大脑已经产生了疑问："这个需求现在什么状态？"
- 但你还需要：
  1. 截图或复制消息
  2. 打开AI工具
  3. 粘贴内容
  4. 输入你的疑问
  5. 等待AI回答
  6. 再去Aone手动查询需求详情
  7. 整理后回复对方

**从"看到问题"到"AI给出答案"，中间隔了7步操作。**

## 我想要的是什么？

**我希望当我看到问题，并且产生疑问或者想法的时候，AI就已经开始工作了。**

而不是需要我再转述我看到的内容和我脑中构思好的想法给AI。

就像移动端的豆包APP，你打开视频通话，AI就能"看到"你周围的环境，你直接问"这是哪里"就能得到答案，而不需要先拍照、上传、再描述场景。

**桌面端也应该有这样的体验。**

# 二、痛点

## 2.1 当前AI Agent的使用痛点

虽然AI Agent功能都很强大，但在企业工作场景中，存在三个核心痛点：

### 痛点1：看不到屏幕，每次都要"转述"

**真实场景：**

**场景A：钉钉消息理解**
```
现状：
1. 看到钉钉消息："今天的实验数据不符合预期"
2. 大脑产生疑问："他具体说的是什么问题？"
3. 截图消息（10秒）
4. 打开ChatGPT（5秒）
5. 上传截图（5秒）
6. 输入："帮我分析这个问题"（10秒）
7. 等待回答（10秒）

耗时：40秒+
痛点：需要转述，打断工作流
```

**场景B：监控数据分析**
```
现状：
1. 看到监控大盘，某个指标飘红
2. 大脑产生疑问："是哪个环节出问题了？"
3. 截图监控（10秒）
4. 打开AI工具（5秒）
5. 上传图片（5秒）
6. 描述问题背景（20秒）
7. 等待分析（15秒）

耗时：55秒+
痛点：复制粘贴，费时费力
```

**场景C：代码Review**
```
现状：
1. 看到一段复杂代码
2. 大脑产生疑问："这段逻辑有没有问题？"
3. 复制代码（5秒）
4. 切换到AI工具（5秒）
5. 粘贴代码（3秒）
6. 输入问题（10秒）
7. 等待回答（10秒）

耗时：33秒+
痛点：频繁切换应用
```

**核心问题：AI看不到你的屏幕，你必须"搬运"信息给它。**

### 痛点2：操作不了企业系统，AI只能"嘴上说"

即使AI看到了屏幕内容，给出了建议，你还是要：

**场景D：Aone需求查询**
```
AI说："这个需求的优先级是P0，建议优先处理"
你问："那关联的子任务有哪些？"
AI说："我看不到Aone系统，你需要手动去查询"

结果：你还是要：
1. 打开Aone（5秒）
2. 搜索需求（10秒）
3. 点击查看（5秒）
4. 手动整理信息（30秒）
5. 回到AI继续对话（5秒）

耗时：55秒+
痛点：AI只能给建议，不能帮你操作
```

**场景E：文档查询**
```
AI说："你可以参考XX技术文档"
你问："能帮我找一下这个文档吗？"
AI说："我没有权限访问钉钉文档，你需要自己去搜索"

结果：你还是要：
1. 打开钉钉文档（5秒）
2. 搜索关键词（10秒）
3. 打开文档（5秒）
4. 阅读和整理（60秒）

耗时：80秒+
痛点：AI无法自动调用企业系统
```

**核心问题：AI操作不了企业系统，所有查询还得你手动完成。**

### 痛点3：信息孤岛，无法基于完整上下文决策

**场景F：综合分析**
```
问题：需要分析一个需求的可行性

需要的信息：
- 屏幕上的需求描述（文档）
- Aone中的需求详情（系统数据）
- 相关的技术文档（知识库）
- 历史类似需求（数据查询）

传统方式：
1. AI看屏幕描述（需要你截图上传）
2. 你手动查Aone（5分钟）
3. 你手动查文档（10分钟）
4. 你手动查历史（5分钟）
5. 你把所有信息整理后再问AI（5分钟）
6. AI基于你整理的信息给建议（1分钟）

耗时：26分钟
痛点：信息分散，需要你当"中间商"
```

**核心问题：AI无法同时获取屏幕内容+企业系统数据，无法基于完整上下文决策。**

## 2.2 移动端已经给出了答案

**豆包APP的启示：**

豆包推出了视频通话功能，你可以：
- 打开摄像头

用户在电脑上工作时：

- 看到数据异常，想知道什么原因

**为什么还要先"转述"给AI，再等答案？**

**桌面端的AI也应该能"看到"你的屏幕，直接基于屏幕内容工作。**

## 2.3 核心痛点总结

让我们算一笔账：

```
传统方式处理一个问题的平均耗时：

每天处理20个类似问题：

- 每月：800-2000分钟 ≈ 13-33小时
```

**这就是我们要解决的核心问题：**

### 4.2.2 MCP协议集成 ⭐ 核心差异化

**什么是MCP？**
- Model Context Protocol（模型上下文协议）
- 由Anthropic（Claude开发商）2024年11月发布
- AI工具调用企业系统的标准化协议
- 让AI从"聊天工具"变成"AI Agent"

**技术实现：**
- 官方SDK：@modelcontextprotocol/sdk v1.25.2
- 传输协议：HTTP/HTTPS + SSE流式
- 认证标准：OAuth 2.1 PKCE
- 安全特性：
  - PKCE S256（防止授权码拦截）
  - State参数（防止CSRF攻击）
  - Resource参数（token与服务器绑定）
  - 自动token刷新
  - 本地加密存储
